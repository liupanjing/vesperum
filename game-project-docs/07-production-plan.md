# 制作计划与执行清单

## 1. 制作策略

本项目应采用“小步验证”的方式。

不要一开始组团队、找画师、写完整世界观、买大量素材。第一阶段最重要的是证明核心循环成立：

> 玩家做选择 -> 显性数值上升 -> 隐藏账本变化 -> NPC 反馈变化 -> Boss 改变 -> 章节结算产生反思。

## 2. 三个月垂直切片计划

### 第 1-2 周：纯文本规则原型

目标：不用引擎，先验证系统逻辑。

任务：

- 整理玻璃镇 NPC 列表。
- 给每个 NPC 添加标签。
- 编写 6-10 个关键选择。
- 定义显性数值。
- 定义隐藏标签账本。
- 编写 Boss 判定规则。
- 用脚本或表格模拟 3 条路线。

产出：

- NPC 表。
- 选择表。
- Boss 规则表。
- 3 条路线测试记录。

验收：

- 同一章节至少能生成 3 个不同 Boss。
- 玩家显性正义值高时，隐藏账本仍可能出现严重负面评价。
- 武力路线能明显改变结算。

### 第 3-4 周：Godot 基础框架

目标：建立可运行项目。

任务：

- 创建 Godot 项目。
- 实现 GameState。
- 实现 ChoiceEngine。
- 实现 PublicLedger。
- 实现基础对话框。
- 实现基础选择菜单。
- 实现 DebugPanel。

产出：

- 可以跑通对话和选择。
- 选择后显性数值和隐藏账本变化。
- DebugPanel 能显示变化。

验收：

- 不改代码即可通过数据新增一个选择。
- NPC 态度能根据标签账本变化。

### 第 5-6 周：基础战斗

目标：卡牌对战可玩，但不追求复杂。

任务：

- 实现小规模卡牌对战。
- 实现主角基础攻击牌。
- 实现 8-12 张基础卡牌。
- 实现 2-4 张武力牌。
- 实现圣剑裁决的剧情入口，但不把圣剑做成普通卡牌。
- 实现 2 个敌人模板。
- 实现卡牌对战胜利后的处置类型。
- 实现武力值变化。

产出：

- 可以从剧情进入卡牌对战。
- 对战后返回剧情。
- 武力值影响后续对话。

验收：

- 玩家能通过战斗解决至少一个冲突。
- 玩家能通过非战斗方式解决至少一个冲突。
- 两种方式后续反馈不同。

### 第 7-9 周：玻璃镇完整章节

目标：完成 Demo 内容闭环。

任务：

- 搭建玻璃镇 2.5D 灰盒场景，包括地上广场、镇议会、地下裂纹区和圣塔入口。
- 加入 8-12 个 NPC。
- 完成主线对话。
- 完成 3-5 个支线。
- 接入 BossResolver。
- 完成 3 个 Boss 分支。
- 完成章节结算。

产出：

- 30-45 分钟可玩 Demo。

验收：

- 至少 3 条路线可通。
- 至少 3 个 Boss 结果可触发。
- 章节结算能显示官方记录和民间记录差异。

### 第 10-11 周：临时美术和音频整合

目标：让 Demo 有基本表现力。

任务：

- 替换关键占位图。
- 加入临时 BGM。
- 加入基础音效。
- 优化 UI。
- 加入主角圣剑阶段变化。
- 加入武力上升后的视觉/音频反馈。

产出：

- 可给外部测试玩家体验的版本。

验收：

- 玩家不需要开发者解释也能完成 Demo。
- 关键选择界面清晰。
- 战斗和结算不会像纯占位。

### 第 12 周：测试与复盘

目标：判断是否继续扩大项目。

任务：

- 找 10-20 名测试玩家。
- 录屏或观察游玩过程。
- 收集问卷。
- 记录卡点。
- 统计路线选择。
- 复盘系统是否成立。

产出：

- 测试报告。
- 下一阶段计划。

验收：

- 玩家理解核心机制。
- 玩家愿意讨论自己的选择。
- 玩家愿意重玩。

## 3. 每周工作节奏

建议一个人开发时采用固定节奏：

- 周一：确定本周目标。
- 周二到周四：实现功能。
- 周五：整合和修 Bug。
- 周六：自己完整跑一遍。
- 周日：记录问题，休息或轻量整理文档。

每周必须有一个可运行版本。哪怕很丑，也要能跑。

## 4. 最小团队配置

### 单人阶段

你负责：

- 引擎学习。
- 系统开发。
- 剧情数据。
- 2.5D 灰盒地图。
- 临时素材整合。

### Demo 美化阶段

建议找：

- 1 名像素/角色画师，按资产包合作。
- 1 名作曲或音效师，按曲目合作。
- 3-5 名早期试玩者。

不建议第一阶段找：

- 全职策划。
- 大量画师。
- 配音演员。
- 市场发行顾问。

## 5. 预算建议

最低预算：

- 引擎：0。
- 素材：0-500 元，使用免费或低价素材。
- 音乐：0-500 元，使用临时授权素材。
- 测试：0。

较舒适 Demo 预算：

- 关键角色立绘：2000-8000 元。
- 关键 Boss 图：1000-5000 元。
- 2.5D 场景模块或地图美化：1000-8000 元。
- BGM 6 首：3000-15000 元。
- 音效包或定制：500-3000 元。

建议先用最低预算做灰盒，确认系统成立后再投入美术和音乐。

## 6. 风险清单

### 范围失控

风险：想做完整世界、完整职业、完整美术。  
应对：第一阶段只做玻璃镇。

### 主题太说教

风险：玩家觉得游戏在训话。  
应对：每个选择都给出真实代价，不提供标准答案。

### 系统太隐晦

风险：玩家完全不知道隐藏账本存在。  
应对：不显示数值，但通过 NPC 称呼、民间记录、价格和支线变化暗示。

### 卡牌对战不好玩

风险：玩家觉得卡牌对战只是剧情门槛。  
应对：卡牌来源必须和调查、显性数值、隐藏账本有关；对战胜利后的处置必须影响剧情。

### 武力路线被误解为坏结局

风险：玩家觉得作者在简单批判强力。  
应对：让武力路线真实改善普通人生活，同时保留反对者和长期制度风险。

### 美术风格不统一

风险：免费素材混用导致廉价。  
应对：灰盒阶段无所谓，公开 Demo 前统一关键视觉。

## 7. 立刻可以做的任务

第一批任务：

- 创建 Godot 项目。
- 建立 Git 仓库。
- 写玻璃镇 NPC 表。
- 写 10 个关键选择。
- 写选择结算 JSON。
- 写一个命令行脚本模拟选择结果。
- 写 Boss 判定规则。

第二批任务：

- 在 Godot 中实现对话框。
- 实现选择菜单。
- 实现显性数值面板。
- 实现 DebugPanel。
- 接入第一批 JSON 数据。

第三批任务：

- 实现基础卡牌对战。
- 接入武力值。
- 接入 BossResolver。
- 做章节结算。

## 8. 测试问卷

Demo 测试后问玩家：

1. 你觉得自己扮演的是怎样的勇者？
2. 哪个选择让你最犹豫？
3. 你是否理解某些 NPC 为什么不认同你？
4. 你是否感受到武力既方便又危险？
5. 你最终遇到的 Boss 是否像是你选择造成的结果？
6. 你愿意重玩一次吗？为什么？
7. 哪里像说教？
8. 哪里不够清楚？
9. 卡牌对战是否有必要？
10. 你最想继续了解哪个角色或地区？

## 9. 下一阶段决策门

12 周 Demo 后，只有满足以下条件才进入第二章开发：

- 核心系统被玩家理解。
- 至少一半测试玩家愿意重玩。
- 开发者能在不大改架构的情况下新增选择和 NPC。
- 玻璃镇内容能在 45 分钟内完成。
- 卡牌对战没有被多数玩家视为纯负担。

如果不满足，先修改 Demo，不要急着扩展世界。
