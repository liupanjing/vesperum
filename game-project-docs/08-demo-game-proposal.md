# Vesperum Demo 游戏企划书

版本：0.1  
状态：立项评审稿  
范围：30-45 分钟垂直切片 Demo  
章节：玻璃镇  
当前阶段：写代码前的企划完善阶段

## 1. 企划目的

本 Demo 不是完整游戏的缩小版，而是用于验证核心设计是否成立的垂直切片。

需要验证的问题：

- 玩家是否能理解“自己眼中的正义”和“群众眼中的勇者”并不一致？
- 玩家是否愿意为自己的选择承担不同群众的评价？
- 武力是否能成为一种有吸引力、有效率、但会改变世界反馈的力量？
- Boss 是否能根据玩家选择变化，而不是固定为所谓邪恶者？
- 这个主题是否能在 30-45 分钟内被玩家感知，而不是靠设定说明灌输？

如果 Demo 不能验证这些问题，完整游戏不应继续扩大制作。

## 2. Demo 核心概念

玩家扮演魔王被讨伐八十年后的白塔巡证官，同时也是第十三号圣剑承载者，来到共和国模范城镇“玻璃镇”，调查圣塔检测到的魔王反应。

在 Demo 中，“勇者”不是主角的正式职位，而是世界投射到主角身上的称呼。官方希望主角像调查员一样按流程确认异常，民间希望主角像神话勇者一样立刻裁决敌人，玩家则要决定自己是否接受这个称呼。

玻璃镇表面零犯罪、高幸福、秩序井然；实际依靠“裂纹者制度”维持美好表象。被判定会拉低幸福指数、制造不稳定的人，会被剥夺正式镇民身份，迁入地下区域继续劳动。

玩家必须在真相、安稳、法律、弱者、普通居民生活和自身武力之间做选择。

Demo 的结尾不固定 Boss。玩家最终面对谁，取决于此前的选择让哪些人受益、受损、恐惧或绝望。

## 3. 玩家体验目标

### 3.1 第一层体验：我被当成勇者

玩家一开始应该感到：

- 自己是来调查异常的。
- 自己拥有合法使命。
- 自己有能力解决问题。
- 大部分 NPC 尊敬或期待“勇者”。

### 3.2 第二层体验：我可能没有理解所有人

中段玩家应开始意识到：

- 同一个选择，对不同人群含义不同。
- 镇长不是单纯坏人。
- 裂纹者不是单纯受害者。
- 普通居民的沉默不是完全无辜，也不是完全邪恶。
- 玩家看见的正义值不能解释所有后果。

### 3.3 第三层体验：我赢了，但这是谁的胜利？

结尾玩家应感到：

- 自己确实解决了一场危机。
- 但某些人并不认为自己被拯救。
- Boss 的出现不是凭空安排，而是自己选择推动的结果。
- 如果重来一次，可能会面对完全不同的敌人。

## 4. 目标平台与形式

Demo 目标：

- 平台：PC 优先。
- 时长：30-45 分钟。
- 视角：2.5D 斜俯视/等距 RPG，不采用横版 2D。
- 表现：立体场景 + 像素角色 + 景深光影 + 对话立绘 + 卡牌对战。
- 操作：键鼠优先，手柄暂不作为首要目标。

2.5D 表现约束：

- 镜头固定或半固定，优先斜俯视/等距构图。
- 场景像一个可探索的立体舞台，不做横版卷轴。
- 角色可以是像素 Sprite 或低成本 3D/2D 混合角色。
- 地上玻璃镇和地下裂纹区必须通过高度、遮挡、灯光和景深形成空间对比。
- Demo 不追求自由旋转视角，优先保证叙事构图和交互可读性。

Demo 不追求：

- 大地图。
- 多章节。
- 复杂装备。
- 大量支线。
- 高规格动画。
- 大规模可旋转 3D 开放场景。
- 完整配音。

## 5. 核心设计支柱

### 5.1 正义不是一个数值能说清的

玩家会看到“正义值”“秩序值”“真实值”等显性数值增长，但这些数值代表的是主角如何理解自己，不代表世界如何评价主角。

### 5.2 群众不是一个整体

NPC 拥有多个标签，例如：

> 男 / 群众 / 穷人 / 农民 / 父亲 / 守序 / 受害者家属

每次选择会影响一批标签，而不是简单影响“好感度”。这让同一个 NPC 可能同时因为玩家的行为获得利益，又因为另一个身份受到伤害。

### 5.3 武力不是坏东西，但会改变人与人的关系

武力可以解决问题。它快速、有效、清晰，甚至可能真的保护多数普通人。

但武力会让 NPC 的反应变形：

- 有人敬畏玩家。
- 有人依赖玩家。
- 有人不再说真话。
- 有人把勇者视为必要的强权。
- 有人把勇者称作新的魔王。

### 5.4 Boss 是选择造成的对立面

Demo 中的 Boss 不预设为邪恶者。

Boss 可能是：

- 镇议会的白玻璃骑士。
- 裂纹者领袖。
- 普通居民恐惧凝成的幸福幻象。
- 圣塔系统生成的秩序裁决体。

玩家不是“找到坏人并打败他”，而是“把某种矛盾推到了必须战斗的程度”。

### 5.5 没有传统任务列表，只有调查档案

Demo 不使用传统 RPG 的任务列表，不显示“下一步去哪里”“找谁交任务”“完成 3/5 个目标”这类指引。

玩家追踪进度的方式是：

- 人物信息。
- 文件日志。
- 事件记录。
- 关系图。
- 玩家自己对线索的判断。

系统只负责记录玩家已经知道的信息，以及这些信息来自谁、是否互相矛盾、与哪些人物或事件有关。系统不替玩家判断什么是真相，也不告诉玩家哪条路线是主线、哪条路线是支线。

## 6. Demo 剧情概要

### 6.1 开场

共和国圣塔检测到玻璃镇出现微弱魔王反应。

玻璃镇是官方宣传中的模范城：

- 零犯罪率。
- 高幸福指数。
- 稳定就业。
- 居民礼貌。
- 没有贫民区。

主角作为白塔巡证官被派往玻璃镇调查。随行者暂定为“法律骑士”，用于代表制度立场，并向玩家解释官方流程。

### 6.2 第一阶段：进入模范城

玩家进入玻璃镇，看到整洁街道、礼貌居民和镇议会欢迎仪式。

玩家接触：

- 镇长。
- 共和国监察官。
- 面包师。
- 巡逻队。
- 普通居民。

第一阶段目标：

- 建立“这里看起来很好”的表层印象。
- 让玩家获得基础操作和对话体验。
- 给玩家第一次显性数值选择。

示例选择：

镇长希望玩家在公开仪式上称赞玻璃镇。

- 称赞玻璃镇的秩序：秩序值上升，官员和普通居民好感上升。
- 保持谨慎，不做评价：真实值上升，官员略不满。
- 当众询问魔王反应：真实值上升，居民不安，监察官警惕。

### 6.3 第二阶段：发现裂纹

玩家发现某些细节不对：

- 居民说话像背过统一话术。
- 某些房子长期空置。
- 夜间有地下运输声。
- 面包师提到“搬走的人”，但立刻改口。
- 圣塔数据与镇议会报告不一致。

玩家可以选择调查路线：

- 走官方路线：查看镇议会档案。
- 走民间路线：询问居民和商人。
- 走地下路线：追踪夜间运输。
- 走武力路线：强行进入封锁区。

不同路线会改变后续 NPC 态度，也会在调查档案中形成不同的信息关系图。

### 6.4 第三阶段：地下裂纹者

玩家进入地下区域，发现“裂纹者”。

裂纹者是被玻璃镇从正式人口中剔除的人。他们继续为镇上提供劳动，但没有完整身份、医疗资格和公共权利。

裂纹者领袖告诉玩家：

- 玻璃镇的零犯罪率来自统计排除。
- 高幸福指数来自样本筛选。
- 地上居民并非全不知道，只是很多人不愿知道。
- 裂纹者内部有人准备发动暴力事件，把真相带到地上。

玩家此时应陷入真实冲突：

- 公开真相可能导致暴乱。
- 压下真相等于再次牺牲裂纹者。
- 强行抓人能维持稳定，但会让勇者变成镇议会的剑。
- 支持裂纹者可能伤害普通居民。

### 6.5 第四阶段：关键选择

Demo 至少提供 4 个核心路线选择。

#### 路线 A：公开真相

玩家公开裂纹者制度证据。

显性变化：

- 正义上升。
- 真实上升。
- 自由上升。

隐藏变化：

- 裂纹者、穷人、被牺牲者对玩家评价上升。
- 官员、守序者、普通居民评价下降。
- 城镇不安上升。

可能 Boss：

- 白玻璃骑士。
- 普通居民恐惧幻象。

#### 路线 B：压下真相，换取补偿改革

玩家暂时隐瞒真相，要求镇议会补偿裂纹者、恢复部分身份。

显性变化：

- 安稳上升。
- 秩序上升。
- 仁慈小幅上升。

隐藏变化：

- 官员、普通居民评价上升。
- 裂纹者、自由派、激进者评价下降。
- 裂纹者领袖可能认为玩家背叛。

可能 Boss：

- 裂纹者领袖。

#### 路线 C：武力威慑镇议会

玩家以勇者权限威胁镇议会交出责任人。

显性变化：

- 正义上升。
- 武力大幅上升。
- 秩序上升或下降，取决于表现。

隐藏变化：

- 官员恐惧上升。
- 普通居民短期安心。
- 恐惧勇者标签上升。
- 部分裂纹者认为玩家有效，部分认为玩家危险。

可能 Boss：

- 幸福幻象。
- 圣塔秩序裁决体。

#### 路线 D：把决定权交给裂纹者

玩家把证据交给裂纹者，让他们自己决定公开方式。

显性变化：

- 自由上升。
- 仁慈上升。
- 真实上升。

隐藏变化：

- 裂纹者评价上升。
- 普通居民命运不确定。
- 镇议会敌意上升。

可能 Boss：

- 镇议会白玻璃骑士。
- 裂纹者内部极端派。

## 7. 主要角色

### 7.1 主角：白塔巡证官 / 第十三号圣剑承载者

定位：玩家化身。  
正式身份：共和国白塔体系的巡证官，负责调查魔王反应。  
特殊身份：第十三号圣剑承载者，拥有启动圣剑裁决的资格。  
公众称呼：勇者。这个称呼来自群众期待、媒体传播和教会叙事，不是主角的职业证书。  
初始状态：相信使命、法律、圣剑和共和国流程。  
Demo 目标：第一次意识到“勇者”并不是自己说了算的身份，而是不同人群基于自身利益和创伤给出的称呼。

### 7.2 法律骑士

定位：随行队友 / 制度解释者。  
标签：骑士 / 守序 / 王都人 / 官员体系 / 荣誉。  
作用：

- 教玩家理解共和国流程。
- 在玩家使用私刑或过度武力时提出质疑。
- 在玩家公开真相导致混乱时提醒秩序代价。

### 7.3 镇长

标签：男 / 官员 / 富人 / 守序 / 既得利益者 / 父亲。  
表面：温和、礼貌、理性。  
真实：知道裂纹者制度，但认为这是保护多数人的必要代价。  
功能：代表安稳与制度理性。

### 7.4 裂纹者领袖

标签：女 / 穷人 / 流民 / 自由 / 被牺牲者 / 激进。  
表面：愤怒，不信任勇者。  
真实：想夺回身份，但也准备牺牲地上居民制造震撼。  
功能：代表被牺牲者的正当愤怒与失控风险。

### 7.5 面包师

标签：男 / 群众 / 平民 / 商人 / 本地人 / 守序 / 旁观者。  
表面：善良、亲切。  
真实：知道一些人被带走，但选择不追问。  
功能：代表普通人的沉默与安稳需求。

### 7.6 地下农夫

标签：男 / 穷人 / 农民 / 父亲 / 被牺牲者 / 受害者家属。  
表面：沉默，害怕勇者。  
真实：孩子因没有身份无法获得治疗。  
功能：让玩家看见“正确政策”落到具体人身上的痛感。

### 7.7 共和国监察官

标签：官员 / 王都人 / 守序 / 记录者 / 既得利益者。  
表面：协助勇者调查。  
真实：希望事件被写成可控事故。  
功能：代表官方叙事如何吸收真相。

## 8. 核心系统设计

### 8.1 显性数值

玩家可见，用于表现主角自我认知。

Demo 使用：

- 正义。
- 仁慈。
- 秩序。
- 自由。
- 真实。
- 安稳。
- 武力。
- 圣剑裁决。

显示方式：

- 角色面板。
- 选择后短提示。
- 章节结算。

注意：显性数值不是客观道德评价。

### 8.2 隐藏群众账本

玩家不可见，用于记录不同标签群体对玩家的真实评价。

Demo 标签池：

- 男。
- 女。
- 穷人。
- 平民。
- 富人。
- 官员。
- 农民。
- 商人。
- 本地人。
- 王都人。
- 守序。
- 自由。
- 激进。
- 旁观者。
- 被牺牲者。
- 既得利益者。
- 受害者家属。
- 恐惧勇者。

隐藏账本影响：

- NPC 对话。
- 是否提供线索。
- 商店或治疗态度。
- Boss 生成。
- 章节结算的民间记录。

### 8.3 武力系统

武力是显性数值，也会写入隐藏账本中的“恐惧勇者”。

武力增加行为：

- 强行闯入。
- 威胁 NPC。
- 拔出圣剑直接裁决或处决关键对象。
- 在卡牌对战中持续使用高压制或高伤害牌。
- 对 Boss 进行处决或公开威慑。

武力反馈：

- 部分 NPC 更配合。
- 部分 NPC 不再说真话。
- 部分冲突可被快速跳过。
- 章节结算中出现“勇者带来秩序”与“勇者带来恐惧”的双重记录。

### 8.4 Boss 生成

Boss 由以下因素决定：

- 关键路线选择。
- 隐藏账本中最低或最高的标签。
- 武力值。
- 是否公开真相。
- 是否压下真相。
- 是否威胁镇议会。
- 是否让裂纹者自主决定。

Boss 不是系统随机，而是规则驱动。

### 8.5 圣剑系统

圣剑是勇者特有的权柄，不作为普通战斗武器，也不进入常规卡牌牌组。

它的功能不是“造成高额伤害”，而是直接结束某个对象的生命或身份。圣剑一旦出鞘，意味着勇者绕过了普通冲突、调查、辩论、审判和战斗过程，直接给出最终裁决。

圣剑可出现的时机：

- 战前：玩家选择不进入卡牌对战，直接处决目标。
- 战中：玩家在卡牌对战失控或陷入僵局时拔剑，强行终止战斗。
- 战后：目标被压制后，玩家选择处决、放过、逮捕或交给他人。
- 剧情节点：面对被系统判定为“魔王反应源”的对象时，玩家可以选择是否执行勇者使命。

圣剑裁决的特点：

- 结果明确，目标死亡或被彻底消除。
- 短期效率极高，能迅速终止危机。
- 武力值和恐惧勇者标签大幅上升。
- 一部分群众会认为勇者果断、可靠、终于解决问题。
- 另一部分群众会认为勇者越过程序，成为不可反抗的暴力。
- 队友、官方、民间记录都会重新解释这次裁决。

关键原则：圣剑不是坏选项，也不是必杀技奖励。它是勇者身份最危险、最不可逆的部分。玩家每次拔剑，都应该感到自己确实拥有解决问题的力量，也确实夺走了其他解决方式存在的可能。

## 9. 战斗系统状态

战斗系统初步改为卡牌对战方向。

上一版“三槽战斗”暂不采用，因为它容易把复杂叙事压成明显 UI 条，降低 RPG 战斗感。

Demo 对战斗系统的要求：

- 需要有真正的卡牌构筑和对局乐趣。
- 需要让武力变强有吸引力。
- 不能把战斗变成单纯惩罚玩家。
- 战斗结果必须影响隐藏账本。
- 战后处置必须有重量。
- 实现成本必须适合单人 Demo。

### 9.1 核心定位

卡牌对战代表普通冲突中的策略、资源、证据、队友、群众支持和个人能力。它解决的是“如何压制、逼退、拖住、说服、瓦解或击败对手”，但不默认杀死对方。

圣剑代表普通冲突之外的绝对裁决。它可以跳过或终止卡牌对战，但会产生更重的政治、道德和群众反馈。

### 9.2 卡牌来源

玩家牌组由几类牌构成：

- 基础行动牌：攻击、防御、闪避、观察、推进。
- 显性数值牌：正义、仁慈、秩序、自由、真实、安稳、武力分别解锁不同风格的牌。
- 证据牌：调查获得，用于削弱 Boss 立场、打断特殊牌或改变战后裁决。
- 队友牌：来自同行角色，代表保护、质疑、支援或反对。
- 群众牌：由隐藏账本间接生成，可能是援助、沉默、阻挠、恐惧配合或背叛。
- 武力牌：高效率、高压制，但每次使用都会增加恐惧勇者相关反馈。

### 9.3 卡牌战斗结果

卡牌对战胜利不等于杀死对方，而是让目标进入“被压制状态”。

被压制后，玩家进入战后处置：

- 逮捕。
- 放逐。
- 赦免。
- 收编。
- 交给群众。
- 交给共和国。
- 拔出圣剑处决。

如果玩家在战前或战中直接拔出圣剑，卡牌对战可以被跳过或中止，但结算会明显偏向武力、恐惧和强力秩序。

### 9.4 Boss 牌组

Boss 牌组由其身份和标签决定：

- 镇议会系 Boss：法律、程序、护卫、封锁、舆论稳定。
- 裂纹者系 Boss：揭露、怒火、牺牲、地下支援、破坏秩序。
- 普通居民幻象：恐惧、沉默、求稳、道德压力、集体否认。
- 圣塔裁决体：封印、审判、净化、权限压制、魔王反应锁定。

Boss 不只是数值更高，而是用牌组表达立场。

### 9.5 仍需定稿的问题

- 卡牌对战是否采用固定小牌组，还是允许玩家构筑？
- Demo 中每场战斗的牌组是否由剧情自动配置？
- 武力牌每次使用是否立刻写入隐藏账本？
- 圣剑是否允许在所有关键战斗中拔出，还是只在特定节点开放？
- 圣剑处决是否需要消耗某种资源，还是只消耗世界对勇者的信任？

## 10. UI 与反馈设计

### 10.1 对话界面

必须清晰显示：

- 说话人。
- 立绘。
- 语气变化。
- 关键概念高亮。

不直接显示隐藏标签。

### 10.2 选择界面

选择文本要表达行动意图，不直接写“正义 +10”。

可以在选择后显示显性变化，例如：

> 正义上升  
> 真实上升  

但不显示隐藏账本变化。

### 10.3 章节结算界面

Demo 最重要的 UI。

建议分为三部分：

#### 勇者记录

玩家可见数值总结：

- 正义。
- 秩序。
- 真实。
- 武力。
- 当前称号。

#### 官方记录

共和国如何记载事件：

> 勇者及时处理玻璃镇异常，防止魔王污染扩散。镇议会将接受内部整改。

#### 民间记录

不同群体如何流传：

> 地上居民说，勇者让他们第一次害怕圣剑。  
> 裂纹者说，勇者终于让他们被看见。  
> 官员说，勇者不懂治理。  

### 10.4 调查档案界面

调查档案取代传统任务列表，是 Demo 中仅次于章节结算的重要 UI。

它由三部分组成：

#### 人物档案

记录玩家已接触或听说过的人物。

每个人物档案包含：

- 姓名或称呼。
- 已知身份。
- 玩家已知标签，但不显示系统隐藏标签全集。
- 与其他人物、势力、文件、事件的关系。
- 玩家听到的关键说法。
- 该人物说法的来源和时间。
- 当前态度的文本化表现，例如“礼貌但回避”“明显恐惧”“愿意协助”“敌意强烈”。

人物档案不直接显示好感度或隐藏账本数值。

#### 文件日志

记录玩家获得或听闻的文件、证言和事实片段。

文件日志包含：

- 官方文件：镇议会报告、圣塔读数、共和国监察记录。
- 民间证言：面包师的说法、地下农夫的请求、裂纹者领袖的指控。
- 现场线索：空置房屋、夜间运输、地下工牌、被涂改的人口表。
- 矛盾记录：同一事件的不同说法。

每条日志必须保留来源。游戏不应直接标注“真/假”，只标注“来源”“相关人物”“相关事件”“与哪些信息冲突”。

#### 关系图

关系图不是任务流程图，而是调查关系图。

节点类型：

- 人物。
- 组织。
- 地点。
- 文件。
- 事件。
- 未解问题。

关系类型：

- 隶属。
- 指控。
- 包庇。
- 受害。
- 获利。
- 目击。
- 矛盾。
- 依赖。
- 恐惧。

示例：

> 镇议会 -> 包庇 -> 裂纹者制度  
> 面包师 -> 目击 -> 夜间运输  
> 地下农夫 -> 受害 -> 身份抹除  
> 圣塔读数 -> 矛盾 -> 镇议会报告  

关系图只展示玩家已经获得的信息，不自动补全真相。玩家需要根据关系图判断接下来要追问谁、查看哪里、相信哪份记录。

### 10.5 不使用传统任务提示

Demo 不显示：

- 当前任务。
- 任务目标。
- 任务完成度。
- 黄色感叹号。
- 地图上的下一步箭头。
- “返回 NPC 交任务”提示。

可以保留弱提示：

- 新信息已加入档案。
- 某人物档案更新。
- 某文件与已有信息冲突。
- 关系图出现新连接。

弱提示只提示“有新信息”，不提示“你应该做什么”。

- 关键场景光照方案。
- UI 框体。

美术关键词：

- 2.5D。
- 斜俯视。
- 立体舞台。
- 像素角色。
- 景深光影。
- 干净。
- 玻璃。
- 白金色。
- 对称。
- 裂纹。
- 地下潮湿。
- 圣洁与压迫并存。

## 12. 音乐需求

Demo 需要 5-6 首临时或定制音乐：

- 标题。
- 玻璃镇地上。
- 地下裂纹区。
- 调查不安。
- Boss 战。
- 章节结算。

音乐目标：

- 前期让玩家感到安稳。
- 中段出现不协调。
- 结尾带来复杂余味。

武力高时，音乐可以加入更低的钟声或军鼓感，但不能变成单纯邪恶主题。

## 13. 大模型使用方案

大模型可以进入本项目，但只能作为表达层工具，不能成为剧情和规则的决策者。

核心原则：

> 系统决定世界发生了什么，大模型只负责世界如何开口。

### 13.1 可用场景

#### 民间传闻

根据隐藏账本、武力值、关键选择和章节状态，生成不同群体对勇者的说法。

示例：

- 官员：勇者及时稳定了玻璃镇局势。
- 裂纹者：勇者又一次让我们闭嘴。
- 普通居民：至少今晚没人再失踪了。
- 孩子：妈妈说不要看勇者的剑。

#### 章节结算文本

章节事实由系统决定，大模型负责生成不同口吻的文本：

- 官方记录。
- 民间记录。
- 地下传单。
- 教会记录。
- 后世史书。

#### 调查档案摘要

大模型可以整理玩家已知信息，帮助玩家回忆目前掌握了什么。

允许：

> 镇议会报告与圣塔读数存在矛盾。面包师目击过夜间运输，但没有说明车上载着什么。

禁止：

> 你下一步应该去地下入口。

#### 关系图说明

关系图节点和边由系统决定，大模型只负责把关系解释成自然语言。

例如：

> 多份记录显示，镇议会至少知道裂纹者制度的存在，但它是否主导了制度仍未确认。

#### 路人短句

路人、守卫、商人、孩子和地下居民的短句可以根据当前状态生成，让玻璃镇对玩家选择产生更细腻的反馈。

#### Boss 开场与战后独白

Boss 身份和事实由系统决定，大模型只负责语气表达。

例如面对高武力玩家时，裂纹者领袖可以说：

> 你不是来听我们说话的，你只是换了一个更亮的地牢。

#### 卡牌命名与风味文本

卡牌效果由系统固定，大模型可以生成名称和描述。同一张武力牌可以在不同路线下被称为“圣令”“肃清”“止乱”或“为了多数人”。

#### 读档回顾

根据玩家已解锁档案生成简短回顾，帮助玩家恢复上下文，但不提示正确答案。

### 13.2 禁止交给大模型的内容

以下内容必须由确定性系统控制：

- 真相判定。
- Boss 生成。
- 隐藏数值变化。
- 卡牌效果。
- 关键选择后果。
- 结局条件。
- NPC 是否知道某个秘密。
- 玩家是否获得某条证据。
- 关系图真实节点和边。

### 13.3 Demo 阶段建议

Demo 阶段优先使用大模型做离线写作辅助：

- NPC 台词草稿。
- 文件日志草稿。
- 民间传闻草稿。
- 章节结算草稿。
- 同一句话的不同情绪版本。

不建议第一版 Demo 直接做全量实时 NPC 生成。

可以做一个小范围实验：

- 只接入 1-2 个非关键 NPC。
- 只生成非关键短句。
- 只能引用已解锁档案。
- 生成结果缓存进存档。
- 不允许生成新事实。

### 13.4 输出约束

如果未来接入实时大模型，输出必须结构化。

示例：

```json
{
  "line": "勇者大人，地下的路不好走。您这样的剑，照不亮那里的。",
  "tone": "fearful_evasive",
  "revealed_facts": ["night_transport_exists"],
  "new_facts": []
}
```

系统必须校验：

- `revealed_facts` 只能来自已解锁事实。
- `new_facts` 必须为空。
- 台词不能改变数值、分支或真相。

## 14. Demo 成功标准

### 14.1 玩家理解

测试玩家应能说出：

- 游戏里有玩家看得见和看不见的两套评价。
- Boss 与玩家选择有关。
- 武力有效，但会改变 NPC 对玩家的态度。

### 14.2 玩家情绪

测试玩家应出现至少一种反应：

- 犹豫。
- 后悔。
- 辩解。
- 想重玩。
- 想知道另一个 Boss 是谁。

### 14.3 开发验证

开发者应能确认：

- 可以通过数据添加新 NPC。
- 可以通过数据添加新选择。
- Boss 规则可调。
- 章节结算能读取状态生成不同文本。

## 15. 制作范围

### 必须做

- 玻璃镇主线。
- 8-12 个 NPC。
- 6-10 个关键选择。
- 人物档案。
- 文件日志。
- 调查关系图。
- 3 个可变 Boss。
- 双层数值系统。
- 武力反馈。
- 章节结算。
- 至少 1 场最终战或最终冲突。

### 可以做

- 1-2 个可选支线。
- 少量场景内演出。
- 简单商店或治疗。
- 队友插话。

### 不做

- 完整装备系统。
- 多队友养成。
- 大地图旅行。
- 复杂解谜。
- 多语言。
- 配音。
- 大量随机卡牌对战。

## 16. 未定问题

以下问题必须在写代码前继续确认：

1. 卡牌对战采用固定小牌组，还是允许玩家在 Demo 内构筑？
2. Demo 是否允许完全不进入卡牌对战通关？
3. 武力值是否只来自卡牌对战，还是也来自威胁、强制调查和圣剑裁决？
4. 玩家是否能看到显性数值具体数字，还是只看到等级/称号？
5. 隐藏账本是否完全隐藏，还是章节结算时部分揭露？
6. 三个 Boss 是否都需要进入卡牌对战，还是允许某些 Boss 被圣剑或事件直接解决？
7. Demo 结尾是否需要保存状态，为未来章节预留？
8. 大模型在 Demo 阶段只做离线写作辅助，还是做小范围实时生成实验？
